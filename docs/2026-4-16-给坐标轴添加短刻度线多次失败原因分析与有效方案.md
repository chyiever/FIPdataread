# 2026-4-16 给坐标轴添加短刻度线多次失败原因分析与有效方案

## 1. 问题背景

本次目标是在 `t-f Plot` 中实现类似刻度尺的短刻度线：

- 左侧频率轴显示主刻度线和子刻度线
- 底部时间轴也显示短刻度线
- 刻度线朝外
- 不把子刻度线画成图内水平虚线/网格线
- 频率轴保持对数坐标，主刻度和子刻度位置正确

实现过程中多次出现“看到了网格线，但没有看到轴边短刻度线”的问题，因此需要单独总结失败原因和最终有效方案。

## 2. 实现过程中遇到的主要问题

### 2.1 误把“轴上的子刻度”实现成了图内水平虚线

最初在 `src/main_window.py` 中，为了让 `t-f` 频率轴显示更多参考线，额外创建了一组 `pg.InfiniteLine`，并把它们加到 `tf_plot` 中。

这带来了两个问题：

- 这些线本质上是绘图区里的横向参考线，不是坐标轴上的短刻度线
- 用户看到的是淡色水平虚线，视觉效果更接近网格，而不是刻度尺

结论：

- `InfiniteLine` 只能用来画图内参考线，不能替代轴刻度线

### 2.2 频率轴子刻度算法写错，导致一个 decade 被等分成 11 格

在 `src/main_window.py` 的 `_update_time_frequency_axis_ticks()` 中，曾使用类似下面的逻辑：

```python
sub_tick = decade + (i / 11.0)
```

这会导致：

- `10 kHz` 到 `100 kHz` 之间被切成 11 个等间隔格
- 子刻度在轴坐标值上等间距
- 视觉上像“线性等分”，不是标准对数子刻度

这也是用户提出“为什么 10kHz 与 100kHz 之间分成了 11 个刻度格，而且还是等间隔”的直接原因。

结论：

- 主轴映射虽然是 `log10(f)`，但子刻度算法并不正确
- 对数轴的标准子刻度应为 `2..9 × 10^n`

### 2.3 误判了 `tickLength` 在左轴上的方向

修改过程中曾假设：

- `tickLength` 为负值时，左轴刻度朝外
- `tickLength` 为正值时，左轴刻度朝里

实际检查 `pyqtgraph.AxisItem` 实现后发现，对左轴来说：

- 正值更接近“朝标签侧/朝外”
- 之前的负值方向判断并不符合当前场景的显示效果

这导致出现了：

- 刻度线方向与预期相反
- 为了修正方向反复调整 `tickLength` 符号

结论：

- 不能只凭经验判断 `tickLength` 方向
- 需要结合 `AxisItem.generateDrawSpecs()` 的具体实现确认左轴和底轴的绘制方向

### 2.4 仅依赖 `setTicks()` 和 `tickLength`，在开启 grid 时无法得到真正的短刻度线

这是本次多次失败的核心原因。

在 `pyqtgraph.AxisItem.generateDrawSpecs()` 中，存在如下关键逻辑：

- 当 `self.grid is False` 时，tick 会被画成短刻度线
- 当 `self.grid` 开启时，tick 的终点会被延伸到 linked view 的边界

换句话说：

- 开启网格时，AxisItem 默认更倾向于把 tick 画成整条网格线
- 用户看到的是贯穿绘图区的水平线/竖线
- 而不是紧贴坐标轴边缘的短刻度线

这解释了为什么多次尝试以下方法都不理想：

- 只改 `tickLength`
- 只改 `maxTickLevel`
- 只改 `maxTextLevel`
- 同时保留 grid 并希望默认 AxisItem 自动给出短刻度

结论：

- 在 `pyqtgraph` 默认行为下，“开启 grid”和“显示独立短刻度线”并不是天然同时成立的
- 如果既要 grid，又要轴边短刻度线，就必须补充自定义绘制逻辑

### 2.5 以为关掉网格后问题已经完全解决，实际上只解决了一半

中途曾采用：

```python
self.tf_plot.showGrid(x=False, y=False)
```

或者：

```python
self.tf_plot.showGrid(x=True, y=False)
```

这类方式确实能避免看到图内水平横线，但本质上只是：

- 取消了干扰现象
- 让默认 tick 更容易被看到

它并没有解决根本问题：

- 一旦重新开启 grid，默认 AxisItem 还是会优先把 tick 延展成网格线

结论：

- 关网格只能缓解现象
- 不能从根本上解决“网格线与短刻度线共存”问题

## 3. 原因分析总结

这次多次失败，不是单一 bug，而是三个层面的误判叠加：

### 3.1 需求对象判断错误

一开始把“轴刻度线”实现成了“图内参考线”，属于对象层级搞错：

- 正确对象：`AxisItem` 上的 ticks
- 错误对象：`PlotItem/ViewBox` 里的 `InfiniteLine`

### 3.2 数学刻度规则错误

频率轴虽然使用 `log10(f)` 作为纵坐标，但子刻度没有按标准对数规则生成，导致视觉上出现错误的等分格。

### 3.3 对第三方库默认行为理解不完整

对 `pyqtgraph.AxisItem` 的理解不够深入，尤其是：

- `grid` 开启时 tick 的绘制行为
- `tickLength` 在不同轴方向上的视觉含义
- `generateDrawSpecs()` 如何生成 tick 与 text 的绘制指令

如果不读实现，只靠调参数，很容易陷入“改了但效果仍然不对”的反复试错。

## 4. 最终有效方案

最终采用了“两层逻辑分离”的方案。

### 4.1 第一层：主/子刻度位置由轴 tick 数据控制

在 `src/main_window.py` 的 `_update_time_frequency_axis_ticks()` 中：

- 主刻度仍按 decade 放置，即 `10^n`
- 子刻度改为标准对数子刻度：

```python
sub_tick = decade + np.log10(factor)   # factor = 2..9
```

这样可保证：

- `1 kHz -> y = 3`
- `10 kHz -> y = 4`
- `100 kHz -> y = 5`

并且：

- `10 kHz` 到 `100 kHz` 之间的子刻度不再等间隔
- 子刻度位置与真实对数坐标一致

### 4.2 第二层：短刻度线由自定义 AxisItem 绘制补充

在 `src/plotting.py` 中新增了两组通用辅助逻辑：

- `_manual_tick_levels()`
- `_append_axis_tick_stubs()`

并在两个轴类中覆写 `generateDrawSpecs()`：

- `AbsoluteTimeAxis`
- `LogFrequencyAxis`

处理方式是：

1. 先调用 `super().generateDrawSpecs(p)`，保留 pyqtgraph 原生的轴线、文字和可能存在的网格线逻辑
2. 再根据 tick levels 额外补画一层“短刻度线”
3. 这层短刻度线紧贴轴边，长度按主/子刻度分级缩短

这一步是本次真正解决问题的关键。

### 4.3 明确区分左轴和底轴的外向方向

在 `_append_axis_tick_stubs()` 中，分别为不同轴设置方向：

- 左轴：从轴边向左画出，表现为朝外
- 底轴：从轴边向下画出，表现为朝外

因此最终效果为：

- 左侧频率轴短刻度线朝外
- 底部时间轴短刻度线朝外

### 4.4 网格策略单独控制，不再让其承担刻度职责

`t-f` 图最终采用：

```python
self.tf_plot.showGrid(x=True, y=False, alpha=0.22)
```

含义是：

- 保留时间方向竖向网格线
- 去掉频率方向水平网格线
- 左轴短刻度线与底轴短刻度线由自定义 AxisItem 单独负责

这样避免了“把网格误认为刻度”的视觉混淆。

## 7. 有效方案的核心结论

本次问题的最终有效方案不是“继续调样式参数”，而是：

1. 使用 `axis.setTicks()` 正确提供主/子刻度数据
2. 对数轴子刻度必须按 `log10(2..9)` 生成
3. 不再用 `InfiniteLine` 模拟轴刻度
4. 阅读并利用 `pyqtgraph.AxisItem.generateDrawSpecs()` 的机制
5. 在 grid 开启场景下，自定义补画轴边短刻度线

只有这样，才能稳定实现以下目标同时成立：

- 刻度位置正确
- 刻度方向正确
- 刻度不是图内横线
- 网格与刻度职责分离
- 左轴和底轴都可复用同一思路

## 8. 后续建议

### 7.1 如果后续还要继续优化外观，优先调以下参数

- 主刻度长度
- 子刻度长度
- 主/子刻度透明度差异
- 轴宽与文字偏移

这些参数都应在当前自定义补绘方案上调整，而不应再退回到 `InfiniteLine` 方案。

### 7.2 如果后续新增右轴或上轴，建议沿用同一框架扩展

当前 `_append_axis_tick_stubs()` 已经是“轴方向感知”的通用框架。若以后需要支持：

- 右轴外向刻度
- 上轴外向刻度

建议继续在该函数中扩展，不要再分别写临时 patch。
